PART 1 — 문제 인식

Postman으로 테스트하면 안 되나요?

API 1번 호출에 수작업 6단계

1. 현재 시각으로 REQ_DTTM 생성
2. TNO 수동 조립
   (ORG_ID + REQ_DTTM + 구분자 + 난수)
3. COMM + DETAIL JSON 직접 작성
4. AES-GCM 암호화 → Base64 인코딩
5. {"DATA": "암호화값"} 형태로 전송
6. 응답 Base64 디코딩 → 복호화

시나리오가 여러 단계일 때

  • MPM: QR 검증 TNO를 다음 단계에 수동 복붙
  • CPM: QR 코드 값을 외부 도구로 이미지 변환
  • 매 테스트마다 동일 반복 → 실수 → 검증 지연
반복 작업의 비효율 + 실수 가능성
→ 자동화된 테스트 도구가 필요

PART 1 — CLAUDE.md

CLAUDE.md — AI의 프로젝트 헌법

## 핵심 규칙
- 서비스 레이어에서 암복호화 처리 금지
  → 반드시 common 모듈에서만 처리
- 공통 클라이언트를 통해서만 외부 API 호출
- Lombok 사용, 네이밍은 camelCase

## 암복호화 스펙
- AES-GCM, APP_CD: B_JBK
- 요청: COMM+DETAIL → JSON → 암호화
         → {"DATA": "BASE64_STRING"}
- 응답: DATA 추출 → 복호화 → JSON 역직렬화

## API 목록
POST /bravo/v1/pay/qr/create  ← QR 생성
POST /bravo/v1/pay/mpm        ← MPM 결제
POST /bravo/v1/pay/status     ← 결제상태

이 파일이 만들어내는 것

  • 모든 구현에서 아키텍처 원칙 자동 준수
  • 대화가 바뀌어도 일관된 코드 스타일 유지
  • 신규 기능 추가 시 기존 패턴을 자동으로 따름
  • 도메인 지식을 한 곳에 집약 → 맥락 재설명 불필요
파일 하나로 AI가 프로젝트의 원칙을 이해

PART 2 — 결제 플로우

MPM 결제: 순서를 지키지 않으면 버튼이 눌리지 않는다

강제된 플로우

1. QR 코드 입력 + 금액 입력
↓
2. QR 검증 → TOKEN + COMM.TNO 저장
↓
3. MPM 결제 버튼 (검증 전 비활성)
↓
4. 결제 상태 조회 (TYPE=M)

자동 주입되는 값

MPM 결제 요청 DETAIL:
  ORG_TNO: [QR검증 COMM.TNO 자동 주입]
  TOKEN:   [QR검증 응답 TOKEN 자동 주입]
  QR_CODE: [입력값 자동 전달]
  UUID:    [선택된 계정 자동 전달]
  AMT:     [입력값]
QR 코드 변경 시 상태 자동 초기화
잘못된 순서 테스트 자체를 UI에서 방지

PART 3 — 활용 사례 ③

버그 진단: 로그를 주면 원인부터 수정까지

발생한 오류

ERROR: Illegal base64 character 7b
  at CryptoUtils.decrypt(line:80)

로그에서 발견:
"RES encrypted: {\"DATA\":\"YAZiV...\"}"
                 ↑
           0x7b = '{' ← JSON 문자!

Claude Code의 진단 과정

1. Read(GatewayResponse.java)
   → body가 DataEnvelope 타입으로 선언

2. 로그 분석
   → 실제 응답: "BODY": "{\"DATA\":\"...\"}"
   → BODY가 객체가 아닌 JSON 문자열!

3. 원인 특정
   → 역직렬화 실패 → null 반환
   → DATA 문자열 전체가 암호화 값으로
     오인 → { 문자에서 base64 오류

4. 수정
   → body 타입을 String으로 변경
   → objectMapper.readValue() 추가
에러 로그만 보여줬을 뿐 — 원인 분석 → 코드 수정 → 커밋까지 자율 처리
PART 3 — 활용 사례 ④

하나를 고치면 다음 문제를 스스로 찾는다

버그 수정 연쇄

버그 ① base64 파싱 오류 → 수정 완료
↓ 다음 오류 발생
버그 ② @Builder만 있고 기본 생성자 없음
→ Jackson 역직렬화 불가
↓ 능동 탐색
버그 ③ AMT/BALANCE String 타입인데
실제 응답은 숫자 0 → 타입 불일치

Claude Code의 대응

TranResultDetailResponse 확인
  → @Builder만 있음

"같은 패턴 다른 파일도 확인"
  → RefundNotiDetailResponse
  → PayRequestDetailResponse
  → CpmCompleteDetailResponse
  → 4개 모두 동일 문제

AMT/BALANCE 타입도 확인
  → String → Long 일괄 수정

ReverseApiService 컴파일 오류 수정
  → Long.parseLong() 변환 적용
같은 패턴의 잠재적 문제를 능동적으로 탐색하여 일괄 수정

아키텍처 원칙 자동 유지

  • URL 설정을 서비스 레이어가 아닌 config 레이어에 배치
  • 런타임 상태는 단일 컴포넌트(EnvConfig)에서 관리
  • UI 토글 → API 호출 → 서버 상태 변경 → 즉시 반영
  • 페이지 로드 시 서버 상태와 UI 자동 동기화

"어떻게 만드는가"에서 "무엇을 만드는가"로 집중점 이동
PART 4 — 한계

AI가 항상 옳지는 않다

실제 발생한 오류들

오류 1: GatewayResponse 구조 오해
  AI가 BODY를 DataEnvelope 객체로 가정
  실제는 JSON 문자열로 인코딩된 형태
  → 런타임 오류 후 수정

오류 2: 배포 환경 추가 의존성 도입
  캐시 수정 시 DEFAULT_ENV 환경변수 추가
  사용자 피드백으로 불필요한 복잡성 인지
  → 제거 후 재배포

반드시 사람이 확인해야 하는 것

  • 보안 민감 코드 — 암복호화 키, 인증 로직
  • 도메인 비즈니스 규칙 — AI가 명세를 오해할 수 있음
  • 운영 환경 배포 — 되돌리기 어려운 작업은 반드시 검토
  • 외부 시스템 연동 — 실제 API가 명세와 다를 수 있음
AI는 강력한 도구이지
판단을 대신하는 존재가 아니다
최종 책임은 언제나 사람에게 있다
PART 4 — 인사이트

Claude Code를 최대한 활용하는 3가지 방법

① CLAUDE.md를 잘 만들어라
  • 아키텍처 규칙 명시 (금지 사항 포함)
  • API 명세 요약 포함
  • 핵심 개념 정의
  • 서버 환경 정보
② 큰 작업은 서브에이전트로
  • 방대한 탐색 → Explore 에이전트
  • 검증 → verifier 에이전트
  • 독립 작업 → 병렬 동시 실행
  • 메인 컨텍스트 절약
③ 피드백을 즉시 제공하라
  • 방향이 잘못되면 바로 수정 지시
  • 구체적일수록 정확하게 수정
  • 맥락 설명보다 결과물 기준으로 지시
  • "~를 유지하면서 ~를 변경"
PART 5 — 정리

Claude Code가 증명한 것

  • 복잡한 도메인 (암복호화 + 결제 플로우)을 처음 접하고 구현 가능
  • 수백 개의 레거시 파일을 자율 탐색하고 업무 흐름 검증 가능
  • 에러 로그를 분석해 원인부터 수정까지 자율 처리 가능
  • 아키텍처 원칙을 스스로 지키며 일관된 코드 생성 가능
  • 사용자 피드백을 받아 즉시 방향 전환 및 재수정 가능
  • Git 커밋, 배포까지 전 과정 자율 수행 가능
AI는 개발자를 대체하지 않는다
AI는 개발자의 생산성을 10배로 만든다
