Background & Goal
어떻게 시작되었나
"API 명세서를 보면서 하나씩 Postman으로 테스트하기엔 흐름이 너무 복잡하다.
시나리오 단위로 테스트할 수 있는 전용 UI가 있으면 어떨까?"
핵심 요구사항
Swagger가 아닌 결제 시나리오 흐름 전체를 UI로 시뮬레이션
CPM(QR스캔) / MPM(QR입력) 두 가지 결제 방식 모두 지원
비플페이가 BK 서버를 호출하는 역방향 API(비플페이 → BK)도 테스트 가능
암복호화 포함된 실제 통신 데이터를 개발자가 눈으로 확인
BK 개발서버와 실제 통신 가능한 외부 배포
Claude Code 주도 개발 — AI와 함께 실무 앱을 만드는 과정 자체를 기록

Troubleshooting ①
코드 레벨 이슈
CASE 01
Jackson 역직렬화 오류
Cannot construct instance of 'SummaryInfo'
(no Creators, like default constructor, exist)
원인: Lombok @Builder만 선언 시 Jackson 역직렬화용 기본 생성자 제거됨
해결: @NoArgsConstructor + @AllArgsConstructor 추가
▸ 에러 로그 붙여넣기 → AI가 원인 분석 + 수정까지 2분 이내 완료
CASE 02
TNO 요청지구분 오적용
역거래 API에 요청지구분 "J" 사용
→ 비플페이 요청 방향은 "B"가 맞음
원인: AI가 역거래 API에도 순방향과 동일하게 J(이용기관) 적용
해결: 개발자가 명세 재확인 후 지적 → B(비플페이)로 즉시 수정
▸ 도메인 세부 규칙은 개발자가 직접 명세서와 대조해야 한다
CASE 03
역거래 API 실제 HTTP 통신 누락
문제: 역거래 테스트 프록시가 BK 외부 서버(dev-gift.appply.co.kr)로 요청하지 않고 내부 서비스를 직접 호출
원인: AI가 역거래 API를 수신 구현으로만 해석
해결: "BK 실제 URL로 HTTP 통신을 내보내야 해"라고 명확히 지시 → RestTemplate 기반 실제 HTTP POST로 재구현
▸ AI는 명시하지 않으면 외부 통신 대신 내부 처리로 구현하는 경향이 있다

Troubleshooting ②
Railway 배포 이슈
CASE 04
암호화 키 누락 — 앱 시작 불가
Could not resolve placeholder 'encryptKey'
in value "${encryptKey}"
원인: crypto.properties가 .gitignore 제외 → Railway 빌드 시 파일 없음
해결: 로컬은 파일 fallback, Railway는 ENCRYPT_KEY 환경변수 fallback 이중 구조 적용
▸ 민감정보는 코드베이스에 절대 포함 않는다는 원칙 유지
CASE 05
Nixpacks — Gradle 9 미지원
Nixpacks build failed
Error: Unsupported Gradle version: 9
원인: Railway 기본 빌더 Nixpacks가 Gradle 9.4.1 미지원
해결: Nixpacks 포기 → Dockerfile 멀티스테이지 빌드로 전환
railway.toml에 builder = "dockerfile" 명시
▸ Dockerfile은 Gradle 버전 제약 없음 → 완전 해결

Efficiency Analysis

작업 항목 기존 방식 AI 협업 단축

Pros, Cons & Takeaways
AI 활용의 장단점
잘 된 것들
✓
반복 구조 코드 — DTO, 컨트롤러, 서비스 패턴이 일관되게 유지
✓
오류 즉시 분석 — 스택트레이스 붙여넣기만 하면 원인 + 수정 2분 이내
✓
문서 → 코드 — API 명세 표를 읽고 DTO, 로직 자동 생성
✓
배포 설정 — Dockerfile, railway.toml 등 낯선 형식도 빠르게 작성
주의가 필요한 것들
!
도메인 세부 규칙 — TNO 요청지구분처럼 명세의 미묘한 차이는 개발자가 검토
!
보수적 해석 — 명시하지 않으면 외부 통신 대신 내부 처리로 구현하는 경향
!
맹신 금지 — 생성된 코드는 반드시 실행 + 도메인 관점 검증 필요
!
보안 민감 정보 — AI에게 키 값 노출 주의, 이중 확인 필수
AI 코딩 어시스턴트는 개발자를 대체하지 않는다.
도메인 지식을 가진 개발자 + AI의 구현 속도 = 강력한 조합
